# ការរួមចំណែក

គម្រោងនេះស្វាគមន៍ការរួមចំណែក និងប្រតិទិន។ ការរួមចំណែកភាគច្រើនត្រូវការឲ្យអ្នកយល់ព្រមប្រកាសសិទ្ធិក្នុងសារៈសម្ព័ន្ធ Contributor License Agreement (CLA) ដែលប្រកាសថាអ្នកមានសិទ្ធិ និងពិតប្រាកដផ្តល់សិទ្ធិដល់យើងក្នុងការប្រើប្រាស់ការរួមចំណែករបស់អ្នក។ សម្រាប់ព័ត៌មានលម្អិត ទស្សនាប្រភេទ [https://cla.opensource.microsoft.com](https://cla.opensource.microsoft.com)

ពេលអ្នកដាក់សំណើ pull request មួយ អ្នកបច្ចេកទេស CLA bot នឹងកំណត់ដោយស្វ័យប្រវត្តិនូវថាតើអ្នកត្រូវផ្តល់ CLA ហើយតែងតាំង PR ដោយសមរម្យ (ឧ. ការត្រួតពិនិត្យស្ថានភាព ការអង្គុយផ្សាយមតិ)។ គ្រាន់តែអនុវត្តតាមសេចក្ដីណែនាំដែលគេផ្តល់ដោយ bot ។ អ្នកត្រូវតែធ្វើបែបនេះតែមួយដងតែម្តងសម្រាប់គេហទំព័រទាំងអស់ដែលប្រើ CLA របស់យើង។

## រឿងនៃអាកប្បកិច្ច

គម្រោងនេះបានអនុម័ត [Microsoft Open Source Code of Conduct](https://opensource.microsoft.com/codeofconduct/) ។ សម្រាប់ព័ត៌មានបន្ថែម អាន [Code of Conduct FAQ](https://opensource.microsoft.com/codeofconduct/faq/) ឬទាក់ទង [opencode@microsoft.com](mailto:opencode@microsoft.com) សម្រាប់សំណួរឬយោបល់បន្ថែម។

## ការព្រមានសម្រាប់ការបង្កើតបញ្ហា

សូមកុំបើកបញ្ហា GitHub សម្រាប់សំណួរឧបត្ថម្ភទូទៅ ព្រោះបញ្ជី GitHub គួរត្រូវបានប្រើសម្រាប់សំណើសំណុំបែបបទ និងរបាយការណ៍កំហុស។ វានឹងអាចធ្វើឲ្យយើងតាមដានបញ្ហាឬកំហុសពិតប្រាកដពីកូដបានកាន់តែងាយស្រួល ហើយថែរក្សាអូស្សារប្រសាសន៍ទូទៅពីកូដពិតប្រាកដឲ្យបែកធ្លាយ។

## របៀបរួមចំណែក

### មគ្គុទេសក៍សំណើ Pull Requests

ពេលដាក់សំណើ pull request (PR) ទៅក្នុងឃ្លាំង Phi-3 CookBook សូមប្រើការណែនាំដូចខាងក្រោម៖

- **Fork the Repository**: តែងតែ fork ឃ្លាំងទៅគណនីផ្ទាល់ខ្លួនរបស់អ្នកមុនធ្វើការផ្លាស់ប្តូរ។

- **Pull requests (PR) ផ្សេងៗ**:
  - ដាក់បម្លែងប្រភេទកែប្រែផ្សេងៗក្នុង pull request ផ្សេងៗ។ ឧ. កែសម្រួលកំហុស និង បច្ចុប្បន្នភាពឯកសារ គួរត្រូវបានដាក់ក្នុង PR ផ្សេងគ្នា។
  - ការកែប្រែ typo និងបច្ចុប្បន្នភាពឯកសារតិចតួចអាចបញ្ចូលជាមួយក្នុង PR តែមួយនៅកន្លែងសមរម្យ។

- **ដោះស្រាយការលំបាកបញ្ចូលខ្លួនទៅគ្នា (Merge conflicts)**៖ បើសំណើ PR របស់អ្នកបង្ហាញភាពបញ្ហាការលំបាកបញ្ចូល ខណៈនេះធ្វើបច្ចុប្បន្នភាព branch `main` ដើម្បីស្របនឹងឃ្លាំងមេមុននឹងធ្វើការកែប្រែរបស់អ្នក។

- **ការដាក់ស្នើការបកប្រែ**៖ ពេលដាក់សំណើ PR បកប្រែ សូមប្រាកដថ Folder បកប្រែមានបកប្រែទាំងអស់សម្រាប់ឯកសាររបស់ Folder ដើម។

### ការណែនាំសរសេរ

ដើម្បីធ្វើឲ្យឯកសារទាំងអស់មានភាពស្របគ្នា សូមប្រើការណែនាំដូចខាងក្រោម៖

- **រាងកំណត់ URL**៖ បញ្ចូល URL ទាំងអស់ក្នុងក្រឡាចត្រង្គបន្ទាត់ ក្រោយពីនោះដាក់ក្នុងក្រោយសញ្ញាក្រឡាចត្រង្គ ទំនេរអ្វីមិនត្រូវមានជុំវិញ ឧ. `[example](https://www.microsoft.com)` ។

- **តំណភ្លើងស្រប**៖ ប្រើ `./` សម្រាប់តំណភ្លើងទាក់ទងទៅឯកសារឬថតនៅថតបច្ចុប្បន្ន ហើយ `../` សម្រាប់ថតផ្ទុកឯកសារនៅថតផ្ទៃខាងលើ។ ឧ. `[example](../../path/to/file)` ឬ `[example](../../../path/to/file)` ។

- **មិនមែន Locale ជាតិសម្បទា**៖ ប្រាកដថាតំណរស្វែងរកមិនមាន locale ជាតិសម្បទា។ ឧ. ចៀសវាង `/en-us/` ឬ `/en/` ។

- **ស្តុករូបភាព**៖ រក្សាទុករូបភាពទាំងអស់នៅក្នុងថត `./imgs`។

- **ឈ្មោះរូបភាពពិពណ៌នា**៖ ឈ្មោះរូបភាពដោយប្រើអក្សរអង់គ្លេស លេខ និងសញ្ញាដាក់ចំណុច។ ឧ. `example-image.jpg` ។

## ការងារជាមួយ GitHub Workflows

ពេលអ្នកដាក់សំណើ pull request វិធីសាស្រ្តខាងក្រោមនឹងត្រូវបើកដំណើរការដើម្បីពិនិត្យការផ្លាស់ប្ដូរ។ អនុវត្តតាមសេចក្ដីណែនាំខាងក្រោមដើម្បីធានាថាសំណើ pull request អ្នកឆ្លងកាត់ការត្រួតពិនិត្យ workflow៖

- [Check Broken Relative Paths](#check-broken-relative-paths)
- [Check URLs Don't Have Locale](#check-urls-dont-have-locale)

### Check Broken Relative Paths

វិធីសាស្រ្តនេះធានាថាតំណភ្ជាប់ស្របក្នុងឯកសាររបស់អ្នកត្រឹមត្រូវ។

1. ដើម្បីប្រាកដថាតំណភ្ជាប់របស់អ្នកដំណើរការបានត្រឹមត្រូវ សូមធ្វើកិច្ចការដូចខាងក្រោមដោយប្រើ VS Code៖
    - លោតលើតំណភ្ជាប់ណាមួយក្នុងឯកសាររបស់អ្នក។
    - ចុច **Ctrl + Click** ដើម្បីបញ្ជូនទៅតំណភ្ជាប់។
    - ប្រសិនបើអ្នកចុចលើតំណភ្ជាប់ហើយវាជួបបញ្ហាក្នុងមូលដ្ឋាន វានឹងបញ្ចេញ workflow និងមិនដំណើរការនៅលើ GitHub ។

1. ដើម្បីជួសជុលបញ្ហានេះ សូមធ្វើកិច្ចការដូចខាងក្រោមដោយប្រើការផ្ដល់អត្ថបទផ្លូវដែលផ្តល់ដោយ VS Code៖
    - វាយ `./` ឬ `../`។
    - VS Code នឹងផ្ដល់ជម្រើសមកសម្រាប់អ្នកជ្រើសរើសតាមអ្វីដែលអ្នកវាយ។
    - អនុវត្តតាមផ្លូវដោយចុចលើឯកសារឬថតដើម្បីធានាថាផ្លូវរបស់អ្នកត្រឹមត្រូវ។

ពេលអ្នកបានបន្ថែមផ្លូវដែលត្រឹមត្រូវ រក្សាទុក ហើយបញ្ចូនការផ្លាស់ប្ដូររបស់អ្នក។

### Check URLs Don't Have Locale

វិធីសាស្រ្តនេះធានាថា URL បណ្តាញមិនមាន locale ជាតិសម្បទា។ ព្រោះឃ្លាំងនេះអាចចូលដំណើរការបានទូទាំងពិភពលោក ម្ចាស់គ្រប់គ្រងត្រូវធានាថា URL មិនមាន locale ប្រទេសរបស់អ្នក។

1. ដើម្បីផ្ទៀងផ្ទាត់ថា URL របស់អ្នកមិនមាន locale ប្រទេស សូមធ្វើកិច្ចការដូចខាងក្រោម៖

    - ស្វែងរកអត្ថបទដូចជា `/en-us/`, `/en/` ឬ locale ភាសាផ្សេងៗក្នុង URL ។

    - ប្រសិនបើអ្វីទាំងនេះមិនមានក្នុង URL របស់អ្នក អ្នកនឹងឈរៀបចំការត្រួតពិនិត្យនេះបាន។

1. ដើម្បីជួសជុលបញ្ហានេះ សូមធ្វើកិច្ចការដូចខាងក្រោម៖
    - បើកផ្លូវឯកសារដែល workflow ខាងលើបានរំលឹក។
    - ដក locale ប្រទេសចេញពី URL ។

ពេលអ្នកដក locale ប្រទេសរួច រក្សាទុក ហើយបញ្ចូនការផ្លាស់ប្ដូររបស់អ្នក។

### Check Broken Urls

វិធីសាស្រ្តនេះធានាថា URL បណ្តាញណាមួយក្នុងឯកសាររបស់អ្នកដំណើរការនិងត្រឡប់ស្ថានភាពកូដ 200 ។

1. ដើម្បីផ្ទៀងផ្ទាត់ថា URL របស់អ្នកដំណើរការត្រឹមត្រូវ សូមធ្វើកិច្ចការដូចខាងក្រោម៖
    - ត្រួតពិនិត្យស្ថានភាពនៃ URL ក្នុងឯកសាររបស់អ្នក។

2. ដើម្បីជួសជុល URL ខូចណាមួយ សូមធ្វើកិច្ចការដូចខាងក្រោម៖
    - បើកឯកសារដែលមាន URL ខូច។
    - បច្ចុប្បន្នភាព URL ទៅតំណភ្ជាប់ត្រឹមត្រូវ។

ពេលអ្នកបានជួសជុល URL រួច រក្សាទុក ហើយបញ្ចូនការផ្លាស់ប្ដូររបស់អ្នក។

> [!NOTE]
>
> អាចមានករណីដែលការត្រួតពិនិត្យ URL បរាជ័យ ទោះបីតំណភ្ជាប់អាចចូលដំណើរការ បាន។ នេះអាចមានហេតុផលដោយសារ៖
>
> - **កំណត់បណ្តាញ:** ម៉ាស៊ីនចល័ត GitHub អាចមានការកំណត់បណ្តាញដែលបញ្ចេញការចូលដំណើរការទៅ URL ពិសេសមួយចំនួន។
> - **បញ្ហាពេលវេលាសម្រាក (Timeout):** URL ដែលយឺតក្នុងការឆ្លើយតបអាចបង្កការត្រួតពិនិត្យ Timeout ក្នុង workflow។
