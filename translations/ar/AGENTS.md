# AGENTS.md

## نظرة عامة على المشروع

PhiCookBook هو مستودع شامل للوصفات يحتوي على أمثلة تطبيقية، ودروس تعليمية، وتوثيقات للعمل مع عائلة Phi من نماذج اللغة الصغيرة (SLMs) التابعة لمايكروسوفت. يعرض المستودع استخدامات متنوعة تشمل الاستدلال، والتخصيص الدقيق، والتكميم، وتنفيذ توليد معزز الاسترجاع (RAG)، والتطبيقات متعددة الوسائط عبر منصات وأطر عمل مختلفة.

**التقنيات الرئيسية:**
- **اللغات:** بايثون، C#/.NET، جافاسكريبت/Node.js
- **الأُطُر:** ONNX Runtime، بايتورش، Transformers، MLX، OpenVINO، Semantic Kernel
- **المنصات:** Microsoft Foundry، نماذج GitHub، Hugging Face، Ollama
- **أنواع النماذج:** Phi-3، Phi-3.5، Phi-4 (نص، رؤية، متعدد الوسائط، أنماط الاستدلال)

**هيكل المستودع:**
- `/code/` - أمثلة كود عاملة وتنفيذات نموذجية
- `/md/` - توثيقات مفصلة، دروس تعليمية وأدلة الاستخدام  
- `/translations/` - ترجمات متعددة اللغات (أكثر من 50 لغة عبر سير عمل مؤتمت)
- `/.devcontainer/` - إعداد حاوية التطوير (بايثون 3.12 مع Ollama)

## إعداد بيئة التطوير

### استخدام GitHub Codespaces أو حاويات التطوير (موصى به)

1. افتح في GitHub Codespaces (الأسرع):
   - اضغط على شارة "Open in GitHub Codespaces" في ملف README
   - يتم تكوين الحاوية تلقائيًا مع بايثون 3.12 وOllama مع Phi-3

2. افتح في حاويات VS Code Dev:
   - استخدم شارة "Open in Dev Containers" من README
   - تتطلب الحاوية ذاكرة مضيف لا تقل عن 16 جيجابايت

### الإعداد المحلي

**المتطلبات:**
- بايثون 3.12 أو أحدث
- .NET 8.0 SDK (لأمثلة C#)
- Node.js 18+ وnpm (لأمثلة جافاسكريبت)
- يُنصح بذاكرة 16 جيجابايت كحد أدنى

**التثبيت:**
```bash
git clone https://github.com/microsoft/PhiCookBook.git
cd PhiCookBook
```

**لأمثلة بايثون:**
انتقل إلى مجلدات الأمثلة المحددة وثبّت التبعيات:
```bash
cd code/<example-directory>
pip install -r requirements.txt  # إذا كان ملف requirements.txt موجودًا
```

**لأمثلة .NET:**
```bash
cd md/04.HOL/dotnet/src
dotnet restore LabsPhi.sln
dotnet build LabsPhi.sln
```

**لأمثلة جافاسكريبت/الويب:**
```bash
cd code/08.RAG/rag_webgpu_chat
npm install
npm run dev  # بدء خادم التطوير
npm run build  # البناء للإنتاج
```

3. **اتباع الدروس** - يجب أن تعمل الأدلة خطوة بخطوة كما هو موثق

**النهج الشائع للتحقق:**
- اختبار تشغيل الأمثلة في البيئة المستهدفة
- التحقق من تثبيت التبعيات بشكل صحيح
- التأكد من تحميل/تنزيل النموذج بنجاح
- مراجعة مطابقة السلوك المتوقع مع التوثيق

## أسلوب الكود والمعايير

### الإرشادات العامة

- يجب أن تكون الأمثلة واضحة، مشروحة، وتعليمية
- اتبع قواعد اللغة الخاصة (PEP 8 لبايثون، معايير C# لـ .NET)
- ركز الأمثلة على إظهار قدرات نماذج Phi المحددة
- أضف تعليقات تشرح المفاهيم الرئيسية والمعلمات الخاصة بالنموذج

### معايير التوثيق

**تنسيق الروابط:**
- استخدم صيغة `[text](../../url)` بدون فراغات زائدة
- الروابط النسبية: استخدم `./` للدليل الحالي، و`../` للأب
- لا تستخدم المناطق الجغرافية أو الوطنية في الروابط (تجنب `/en-us/`، `/en/`)

**الصور:**
- خزّن جميع الصور في دليل `/imgs/`
- استخدم أسماء وصفية بالحروف الإنجليزية، الأرقام والواصلات
- مثال: `phi-3-architecture.png`

**ملفات ماركدوان:**
- أشر إلى الأمثلة العملية داخل `/code/`
- حافظ على تزامن الوثائق مع تغييرات الكود
- استخدم الرمز 📓 لوضع علامة على روابط دفاتر Jupyter في README

### تنظيم الملفات

- الأمثلة في `/code/` منظمة حسب الموضوع/الميزة
- التوثيقات في `/md/` تعكس هيكل الكود عند الإمكان
- اجمع الملفات ذات الصلة (دفاتر، سكريبتات، إعدادات) في مجلدات فرعية

## إرشادات طلبات السحب (PR)

### قبل الإرسال

1. **افتح نسخة (fork) من المستودع** إلى حسابك
2. **افصل طلبات السحب حسب النوع:**
   - إصلاحات الأخطاء في طلب منفصل
   - تحديثات التوثيق في طلب آخر
   - أمثلة جديدة في طلبات منفصلة
   - يمكن دمج تصحيحات الأخطاء الإملائية

3. **تعامل مع تعارضات الدمج:**
   - حدّث فرع `main` المحلي قبل التغييرات
   - ساهم بتزامن متكرر مع الأصل

4. **طلبات الترجمة:**
   - يجب أن تتضمن ترجمات لجميع الملفات في المجلد
   - حافظ على نفس الهيكل كما في اللغة الأصلية

### الفحوصات المطلوبة

تعمل سير عمل GitHub تلقائيًا على التحقق من:

1. **التحقق من المسارات النسبية** - جميع الروابط الداخلية يجب أن تعمل
   - اختبر الروابط محليًا: Ctrl+Click في VS Code
   - استخدم اقتراحات المسارات في VS Code (`./` أو `../`)

2. **التحقق من موقع URL** - يجب ألا تحتوي روابط الويب على مناطق بلد
   - احذف `/en-us/`، `/en/` أو أكواد لغات أخرى
   - استخدم روابط عالمية عامة

3. **التحقق من الروابط المعطوبة** - كل الروابط يجب أن تُرجع حالة 200
   - تحقق من الوصول للروابط قبل الإرسال
   - ملاحظة: بعض الفشل يمكن أن يكون بسبب قيود الشبكة

## أنماط التطوير الشائعة

### العمل مع نماذج Phi

**تحميل النموذج:**
- تستخدم الأمثلة أطر عمل متعددة: Transformers، ONNX Runtime، MLX، OpenVINO
- يتم عادة تنزيل النماذج من Hugging Face، Azure أو GitHub Models
- تحقق من توافق النموذج مع عتادك (CPU، GPU، NPU)

**أنماط الاستدلال:**
- توليد النص: معظم الأمثلة تستخدم متغيرات الدردشة/التوجيه
- الرؤية: Phi-3-vision وPhi-4-multimodal لفهم الصور
- الصوت: Phi-4-multimodal يدعم إدخالات الصوت
- الاستدلال: متغيرات Phi-4-reasoning للمهام الاستدلالية المتقدمة

### ملاحظات مخصصة للمنصات

**Microsoft Foundry:**
- يتطلب اشتراك Azure ومفاتيح API
- راجع `/md/02.QuickStart/AzureAIFoundry_QuickStart.md`

**نماذج GitHub:**
- متوفر طبقة مجانية للاختبار
- راجع `/md/02.QuickStart/GitHubModel_QuickStart.md`

**الاستدلال المحلي:**
- ONNX Runtime: متعدد المنصات، استدلال محسن
- Ollama: إدارة نماذج محلية سهلة (مهيأة مسبقًا في الحاوية)
- Apple MLX: محسن لشرائح Apple Silicon

## استكشاف الأخطاء وإصلاحها

### المشكلات الشائعة

**مشكلات الذاكرة:**
- نماذج Phi تحتاج ذاكرة رام كبيرة (خاصة الرؤية/متعدد الوسائط)
- استخدم نماذج مكممة للبيئات محدودة الموارد
- راجع `/md/01.Introduction/04/QuantifyingPhi.md`

**تعارضات التبعيات:**
- قد يكون لأمثلة بايثون متطلبات إصدار محددة
- استخدم بيئات افتراضية لكل مثال
- تحقق من ملفات `requirements.txt` الفردية

**فشل تنزيل النموذج:**
- النماذج الكبيرة قد تنقطع على اتصالات بطيئة
- فكر باستخدام بيئات سحابية (Codespaces، Azure)
- تحقق من ذاكرة التخزين المؤقت لـ Hugging Face: `~/.cache/huggingface/`

**مشكلات مشاريع .NET:**
- تأكد من تثبيت .NET 8.0 SDK
- استخدم `dotnet restore` قبل البناء
- بعض المشاريع لديها إعدادات مخصصة لـ CUDA (Debug_Cuda)

**أمثلة جافاسكريبت/الويب:**
- استخدم Node.js 18+ للتوافق
- امسح مجلد `node_modules` وأعد التثبيت إن استمرت المشكلات
- تحقق من وحدة تحكم المتصفح لمشكلات توافق WebGPU
